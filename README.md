# **TORA: Transparent Orchestration of ReAct Agents**  
*A minimal architecture for transparent action selection in LLM agents*

---

## **Overview**

TORA (Transparent Orchestration of ReAct Agents) is a minimal and general architecture designed to solve a structural limitation of ReAct-style LLM agents:

> **ReAct agents cannot explain *why* they selected a particular action.**

While ReAct exposes the *reasoning text* generated by the LLM, the actual decision-making process remains opaque:

- Why that reasoning was chosen  
- Why a specific tool was selected  
- What alternative actions were considered  

TORA addresses this by **externalizing the action-selection layer** and making it fully transparent, while keeping ReAct agents themselves as black boxes.

This repository contains a minimal implementation of **TASL (Transparent Action Selection Layer)**, the core component of TORA.

---

## **Key Idea**

TORA separates:

- **Action Selection → Transparent**  
- **Action Execution → Black Box**

This mirrors human decision-making:

- We can explain *why* we chose an action  
- But we cannot explain neural-level details of *how* our body executes it

TORA brings this structure to LLM agents.

---

## **Architecture**

TORA consists of four minimal components:

### **1. Action Scenario List**
- Human-defined objectives written in natural language  
- One scenario = one clear purpose  
- Each scenario maps 1:1 to a ReAct agent  

### **2. Scenario Selector (TASL)**
- A transparent decision-making module  
- Rule-based, scoring, utility maximization — anything is allowed  
- Outputs: selected scenario + rationale  

### **3. ReAct Executor**
- Executes the ReAct agent corresponding to the selected scenario  
- Internal reasoning remains a black box  
- In this implementation, ReAct agents are intentionally **empty stubs**  

### **4. Trace Logger**
- Logs:  
  - selected scenario  
  - rationale  
  - compared candidates  
  - executed agent  
  - execution result  

This produces a fully transparent action-selection trace.

---

## **Why TORA?**

Existing approaches fail to provide transparency:

| Method | Limitation |
|--------|------------|
| **ReAct** | reasoning is visible, but *action selection* is opaque |
| **Multi-agent systems** | opacity compounds across agents |
| **Multi-tool LLMs (MCP, Toolformer)** | tool selection rationale is hidden |
| **XAI** | interprets internal reasoning, not action selection |

TORA is the **first architecture** that directly targets the transparency of the action-selection layer.

## **Minimal Example (from the paper)**

### **Action Scenarios**
- `escape` — move away from danger  
- `scan` — detect nearby threats  
- `explore` — explore surroundings  

### **Rule-Based Selector**
- danger ≥ 7 → escape  
- enemy detected → scan  
- otherwise → explore  

### **ReAct Agents**
- EscapeAgent  
- ScanAgent  
- ExploreAgent  

These agents are intentionally implemented as **empty black-box stubs**,  
because TORA focuses on *selection*, not *execution*.

## **Quick Start**

This repository contains a minimal implementation of **TORA/TASL**.  
No dependencies are required.

---

### **1. Run the minimal demo**

```bash
python tasl.py
```

Example output:

```
State: danger=8, enemy_detected=True
Selected Scenario: escape
Reason: Danger level is high (8 >= 7)
ReAct Executor: EscapeAgent
```

This demonstrates:

- scenario enumeration  
- transparent decision-making  
- rationale logging  
- execution of the corresponding (black-box) ReAct agent  

---

### **2. Modify the decision rules**

The Scenario Selector is intentionally simple and fully transparent.

In `main.py`, you can freely modify the rule-based logic:

```python
if state["danger"] >= 7:
    return "escape", "Danger level is high"
elif state["enemy_detected"]:
    return "scan", "Enemy detected"
else:
    return "explore", "No danger or enemy detected"
```

You can replace this with:

- scoring functions  
- utility maximization  
- weighted rules  
- human-defined value models  
- or any transparent algorithm you prefer  

As long as the logic is **transparent**,  
it remains compliant with the TORA/TASL architecture.

---

### **3. ReAct Agents are intentionally empty**

ReAct agents are implemented as **empty stubs**, e.g.:

```python
class EscapeAgent:
    def run(self, state):
        return {"result": "escape executed"}
```

This reflects TORA’s core principle:

> **“Action selection is transparent; action execution remains a black box.”**

You may extend these agents with real reasoning or tool use,  
but this is not required for TORA to function.

---

### **4. Inspect transparency logs**

Each run prints:

- the selected scenario  
- the rationale  
- the compared candidates  
- the executed agent  
- the execution result  

This provides a complete, human-readable trace of the action-selection process.
